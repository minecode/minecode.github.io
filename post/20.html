REST e RESTful - Parte 2

Posted on April, 2020 byfabiohfab

Na primeira parte deste artigo foram abordadas as normas de uma API REST e foi feita uma abordagem aos métodos HTTP. É por isso recomendada a leitura para os leitores que não estejam familiarizados com estes tópicos.

Quando um utilizador da API solicita alguns recursos específicos, por exemplo, recursos que tenham de ser filtrados por localização geográfica, não faz sentido enviar na resposta da solicitação todos os recursos, deixando a cargo do cliente a filtração dos mesmos.

Nesta abordagem enviar-se-ia um grande conjunto de dados em que muitos deles não seriam úteis para o cliente. Por exemplo, ao fazer a seguinte requisição, a API irá retornar todos os restaurantes que tem em base de dados. Caso o utilizador apenas necessite dos restaurantes localizados em Aveiro, terá de ser ele a filtrar os resultados.

REQUEST:
GET /restaurant/

RESPONSE:
[
    {"id": 1, "name": "asd", "location": "Porto"},
    {"id": 2, "name": "aaa", "location": "Lisboa"},
    {"id": 3, "name": "ccc", "location": "Lisboa"},
    {"id": 4, "name": "bbb", "location": "Coimbra"},
    ...,
    {"id": 100, "name": "abc", "location": "Aveiro"},
    {"id": 101, "name": "123", "location": "Faro"},
    ...
]

Ao implementar a filtração do lado da API, a requisição poderá ser feita da seguinte forma:

REQUEST:
GET /restaurant/?location=Aveiro

RESPONSE:
[
    {"id": 35, "name": "xyz", "location": "Aveiro"},
    {"id": 100, "name": "abc", "location": "Aveiro"}
]

Assim sendo, será mais aconselhável fazer a filtração de resultados do lado da API e posteriormente enviar esses resultados para o cliente. A ordenação pode também ser bastante útil, caso o utilizador da API necessite saber, por exemplo, os menus mais baratos de um restaurante. 

REQUEST:
GET /restaurant/2/?sort=-price

RESPONSE:
[
    {"id": 20, "restaurant": 2, "name": "beef", "price": "50"},
    {"id": 21, "restaurant": 2, "name": "fish", "price": "45"},
    {"id": 22, "restaurant": 2, "name": "vegetarian", "price": "30"}
]

Mesmo filtrando os resultados e ordenando-os consoante as necessidades do utilizador, a requisição poderá enviar muitos recursos na sua resposta, o que poderá causar alguns problemas de interpretação dos resultados por parte do utilizador. A melhor solução a este problema será a paginação, ou seja, enviar os resultados em conjuntos de x elementos. 

REQUEST:
GET /restaurant/?page=2

RESPONSE:
[
    {"id": 5, "name": "asd", "location": "Porto"},
    {"id": 6, "name": "aaa", "location": "Lisboa"},
    {"id": 7, "name": "ccc", "location": "Lisboa"},
    {"id": 8, "name": "bbb", "location": "Coimbra"}
]

No exemplo em cima apenas são retornados 4 restaurantes registados na aplicação, embora existam mais.

Para que uma API seja bem definida, é necessário que o seu utilizador saiba o que pode ou não fazer. O HATEOAS, Hypermedia as the Engine of Application State, permite "documentar" a API através de links que redirecionam o cliente para as possíveis ações que o utilizador pode fazer em cada recurso. Por exemplo, no caso anterior da paginação, seria útil enviar na resposta da solicitação os links para a próxima página e para a anterior. Outras informações deverão ser adicionadas à resposta, tais como, o número de recursos enviados na mesma, o número total de páginas e a página atual. No caso da resposta ser uma lista de recursos, poderão ser adicionados links para o acesso direto a cada recurso, para o editar, apagar, etc.. Desta forma, em cada solicitação, a resposta daria ao cliente as possíveis ações para cada recurso.

Note-se que é importante que as respostas das solicitações sejam simples, fáceis de entender mas que ao mesmo tempo tenham toda a informação necessária e útil. Por isso mesmo é muito importante a escolha do tipo de média. Um dos problemas do JSON é que não suporta hypermedia, ou seja, não segue totalmente uma ideologia RESTful. No caso de se usar JSON, a melhor solução será enviar a informação solicitada dentro de uma key results e os links para outros recursos e ações em metadata fora dos results. Ainda assim será necessário incorporar alguns links em cada recurso, o que não é o ideal pois irá misturar links com dados de recursos.

Posto isto, a requisição que vimos anteriormente ficaria da seguinte forma:

REQUEST:
GET /restaurant/?page=2

RESPONSE:
{
    "page_number": 2,
    "total_pages": 102,
    "total_count": 1013,
    "previous_page": "/stations/?page=1",
    "next_page": "/stations/?page=3",
    "last_page": "/stations/?page=102",
    "results": [
        {"self": "/restaurant/5/", "id": 5, "name": "asd", "location": "Porto"},
        {"self": "/restaurant/6/", "id": 6, "name": "aaa", "location": "Lisboa"},
        {"self": "/restaurant/7/", "id": 7, "name": "ccc", "location": "Lisboa"},
        {"self": "/restaurant/8/", "id": 8, "name": "bbb", "location": "Coimbra"}
    ]
}

A alternativa mais RESTful ao JSON será usar tipos de média com suporte direto ao Hypermedia, tais como o API JSON. Ao utilizar esta alternativa, iremos perder alguma da simplicidade de leitura e escrita das respostas das solicitações, pelo que terá de ser feita uma escolha que melhor se adeque ao que é pretendido.
