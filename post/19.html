REST e RESTful - Parte 1

Posted on April, 2020 byfabiohfab

Uma API RESTful é uma API que segue um conjunto de normas de arquitetura REST. Esta abordagem permite que a API seja simples, extensível, confiável e que tenha um bom desempenho.

Uma API que seja simples significa que a forma como se interage com os recursos desta é bem definida e rigorosa. As interações são "sem estado" ou stateless, o que implica que todas as solicitações devem conter a informação necessária para executar determinado recurso. A extensibilidade permite que a API tenha a capacidade de representar os mesmos recursos em diferentes formatos ou versões e que seja capaz de adicionar novos recursos sem afetar outros já existentes. A confiabilidade introduz na API uma separação clara de ações com e sem efeitos colaterais. Seguidamente são apresentadas algumas abordagens que ajudam a implementar estes conceitos.

No estilo arquitetónico REST, existem alguns elementos e termos básicos que devem ser conhecidos.

Quando se fala em recursos, refere-se aos "substantivos" do sistema. Por exemplo, numa API de entrega de comida, os recursos seriam restaurante, menu, etc..

As representações são a forma como os clientes da API vêem os recursos. Uma API Restful nunca deverá permitir a um utilizador aceder diretamente aos recursos. As iterações devem acontecer através de representações dos recursos. As ações são muito úteis porque, visto que os usuários não podem aceder diretamente aos recursos, são necessárias ações para que os possam alterar. As ações são os "verbos" do sistema. Os URLs são a forma como conseguimos localizar os recursos na internet. Os tipos de média especificam como os dados devem estar padronizados nas solicitações e nas respostas. Como já vimos anteriormente, os mais comuns de se usar são o JSON, XML e YAML. Os métodos, métodos internos de HTTP, são a forma como se implementa as diferentes ações REST cujos mais comuns são o GET, POST, PUT e DELETE. Os métodos são chamados por URLs e possuem dados representados por tipos de média. O GET, tal como o nome indica, deve ser usado para recuperar dados do servidor. Para podermos alterar recursos devem ser usados os seguintes métodos, consoante as necessidades. O POST para introduzir novas informações no servidor. Essas informações devem ser passadas no coroo da solicitação de acordo com o tipo de mídia escolhido.O PUT é em todo semelhante ao POST com a diferença que deve ser usado para atualizar informações de um determinado recurso. Por fim, o DELETE deverá ser usado para remover recursos do servidor.

Um passo importante na construção de APIs RESTful é a identificação dos recursos e das ações da especificação do sistema e, posteriormente, extrair os URLs e os métodos aos quais eles respondem a partir desses recursos e ações.

Como falado anteriormente, as APIs RESTful devem ser bem definidas e rigorosas, não restando dúvidas de que determinada ação foi cumprida com sucesso. Por isso mesmo, existem códigos de estado HTTP que devem ser implementados nas respostas às solicitações.

De forma geral, as respostas podem conter códigos com o formato:

  • 1xx: mensagem apenas informativas;
  • 2xx: a solicitação obteve sucesso e não ocorreram falhas;
  • 3xx: redirecionamento da solicitação;
  • 4xx: erros do cliente, por exemplo, o corpo da solicitação tem um formato errado;
  • 5xx: falha ocorrida no lado do servidor.

Nos dois últimos casos, para além do código de estado, também devem ser retornados na resposta procedimentos ou informações úteis para que a falha seja resolvida.

Outros termos que devem ser tidos em conta quando se fala em simplicidade e performance da aplicação são a ordenação, filtração, paginação e links.
